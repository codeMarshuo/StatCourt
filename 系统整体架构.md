# StatCourt NBA数据统计平台 - 系统整体架构

## 一、架构概述

StatCourt NBA数据统计平台采用**前后端分离架构**，基于**B/S模式**设计，遵循**MVC设计模式**，将系统划分为表现层、业务逻辑层、数据访问层和数据层四个层次。

### 1.1 架构特点

| 特点 | 说明 |
|------|------|
| **前后端分离** | 前端负责界面展示，后端提供API服务 |
| **分层架构** | 各层职责明确，降低耦合度 |
| **RESTful设计** | 统一接口规范，便于前后端协作 |
| **可扩展性** | 模块化设计，支持功能扩展 |

## 三、分层架构详解

### 3.1 表现层 (Presentation Layer)

**职责**：处理HTTP请求，返回响应结果

**组件**：
| 控制器 | 功能 |
|--------|------|
| AuthController | 用户认证（登录、注册、个人信息管理） |
| DataController | 数据查询（球员、球队数据） |
| AIController | AI对话（智能问答、流式响应） |
| PredictionController | 比赛预测（AI预测、结果分析） |
| AdvancedStatsController | 高阶数据查询和对比 |

**技术实现**：
- Spring MVC @RestController注解
- RESTful API设计
- SSE流式响应支持

### 3.2 业务逻辑层 (Business Logic Layer)

**职责**：实现核心业务逻辑

**组件**：
| 服务类 | 功能 |
|--------|------|
| DataService | 数据查询业务逻辑 |
| AIService | AI对话业务逻辑 |
| SearchService | 联网搜索业务逻辑 |
| BasketballPredictionService | 比赛预测业务逻辑 |

**技术实现**：
- Spring @Service注解
- 依赖注入管理
- 事务控制

### 3.3 数据访问层 (Data Access Layer)

**职责**：封装数据库操作

**组件**：
| Mapper接口 | 功能 |
|------------|------|
| EmpMapper | 用户数据访问 |
| UserStatsMapper | 用户统计数据访问 |
| PlayerMapper | 球员数据访问 |
| TeamMapper | 球队数据访问 |
| AdvancedStatsMapper | 高阶数据访问 |

**技术实现**：
- MyBatis @Mapper注解
- SQL注解方式定义
- 结果集自动映射

### 3.4 数据层 (Data Layer)

**职责**：数据持久化存储

**组件**：
| 数据库表 | 存储内容 |
|----------|----------|
| emp | 用户基本信息 |
| user_stats | 用户统计数据 |
| nbaplayerbasicstatistics | 球员基础数据 |
| nbateamstats | 球队数据 |
| nba_advanced_stats | 球员高阶数据 |

**技术实现**：
- MySQL 8.0关系型数据库
- InnoDB存储引擎
- utf8mb4字符集
