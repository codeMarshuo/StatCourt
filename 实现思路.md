# StatCourt NBA数据统计平台 - 实现思路

## 一、系统设计思路

### 1.1 整体设计理念

本系统采用**前后端分离架构**，遵循**MVC设计模式**，将系统划分为表现层、业务逻辑层、数据访问层和数据层四个层次。通过分层设计实现高内聚低耦合，便于系统维护和扩展。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **模块化设计** | 各功能模块独立开发，降低耦合度 |
| **可扩展性** | 预留接口扩展空间，支持功能迭代 |
| **用户友好** | 注重交互体验，界面简洁美观 |
| **性能优先** | 采用缓存、异步等技术提升响应速度 |

## 二、技术选型思路

### 2.1 后端技术选型

**选择Spring Boot的原因：**
- 简化配置，开箱即用，提高开发效率
- 内置Tomcat服务器，无需部署WAR包
- 丰富的Starter依赖，快速集成各种技术
- 强大的社区支持，问题易于解决

**选择MyBatis的原因：**
- SQL灵活可控，便于复杂查询优化
- 支持注解和XML两种配置方式
- 与Spring Boot无缝集成
- 学习成本低，上手快

**选择MySQL的原因：**
- 开源免费，降低项目成本
- 性能稳定，满足数据存储需求
- 社区活跃，资料丰富
- 支持事务，保证数据一致性

### 2.2 前端技术选型

**选择ECharts的原因：**
- Apache开源项目，稳定可靠
- 图表类型丰富，满足各种可视化需求
- 中文文档完善，易于学习
- 交互性强，支持动画效果

**选择Three.js的原因：**
- WebGL封装，无需插件即可实现3D效果
- 社区活跃，示例丰富
- 性能优秀，渲染流畅
- 支持多种3D模型格式

**选择Leaflet的原因：**
- 轻量级，体积小加载快
- API简单，易于集成
- 插件生态丰富
- 移动端兼容性好

### 2.3 AI技术选型

**选择智谱AI的原因：**
- 国产大模型，中文理解能力强
- API接口友好，调用简单
- 支持流式响应，用户体验好
- 价格合理，适合学生项目

## 三、核心模块实现思路

### 3.1 用户管理模块

**实现思路：**

1. **注册功能**：前端验证用户名和密码格式，后端检查用户名唯一性，使用MyBatis插入用户记录，同时创建用户统计记录。

2. **登录功能**：验证用户名密码，生成UUID作为Token，存储在内存Map中，返回给前端保存在localStorage，实现会话保持。

3. **个人中心**：通过Token获取用户信息，支持修改昵称、密码、设置喜爱球队和球员，更新操作同步到数据库。

**关键代码逻辑：**
```java
// 登录验证流程
Emp emp = empMapper.findByUsername(username);
if (emp != null && emp.getPassword().equals(password)) {
    String token = UUID.randomUUID().toString();
    tokenStore.put(token, emp);
    return Result.success(Map.of("token", token, "user", emp));
}
```

### 3.2 数据查询模块

**实现思路：**

1. **数据获取**：通过MyBatis Mapper接口查询MySQL数据库，获取球员和球队数据。

2. **数据展示**：前端使用Axios发送请求，接收JSON数据后渲染到表格组件。

3. **筛选排序**：前端实现本地筛选和排序功能，减少服务器压力。

4. **详情查看**：点击球员或球队，发送请求获取详细信息，展示在弹窗或新页面。

**关键代码逻辑：**
```java
// 数据查询流程
@GetMapping("/players")
public Result<?> getAllPlayers() {
    List<Player> players = playerMapper.findAll();
    return Result.success(players);
}
```

### 3.3 数据可视化模块

**实现思路：**

1. **2D图表**：使用ECharts初始化图表实例，配置option对象设置图表类型、数据和样式，调用setOption方法渲染。

2. **3D可视化**：使用Three.js创建场景、相机、渲染器，添加几何体和材质，通过动画循环实现自动旋转。

3. **球队地图**：使用Leaflet初始化地图，设置球队坐标，创建自定义标记，点击标记显示球队详情弹窗。

**关键代码逻辑：**
```javascript
// ECharts图表初始化
const chart = echarts.init(container);
chart.setOption({
    title: { text: '球员数据雷达图' },
    radar: { indicator: indicators },
    series: [{ type: 'radar', data: playerData }]
});
```

### 3.4 AI对话模块

**实现思路：**

1. **SSE连接**：前端创建EventSource对象，建立与服务器的长连接，监听消息事件。

2. **请求处理**：后端创建SseEmitter对象，设置超时时间，通过ExecutorService异步处理请求。

3. **联网搜索**：调用SearchAPI获取实时信息，将搜索结果作为上下文添加到提示词。

4. **AI调用**：使用OkHttp发送POST请求到智谱AI API，设置stream=true启用流式响应。

5. **流式返回**：解析SSE数据流，提取content字段，通过SseEmitter发送给前端实时渲染。

**关键代码逻辑：**
```java
// SSE流式响应
SseEmitter emitter = new SseEmitter(180000L);
executorService.execute(() -> {
    // 调用AI API
    // 解析响应
    // 发送SSE事件
    emitter.send(SseEmitter.event().name("content").data(content));
    emitter.complete();
});
return emitter;
```

### 3.5 比赛预测模块

**实现思路：**

1. **数据准备**：从数据库获取两队的战绩、场均数据等信息。

2. **提示词构建**：将球队数据格式化为JSON，构建包含分析要求的提示词。

3. **AI预测**：调用智谱AI API，要求返回JSON格式的预测结果。

4. **结果解析**：解析AI返回的JSON，提取胜负概率、分差预测等数据。

5. **降级处理**：当API限流时，使用本地算法基于胜率计算预测结果。

**关键代码逻辑：**
```java
// 构建预测提示词
String prompt = String.format(
    "分析以下两队比赛：\n主队：%s（胜率%s）\n客队：%s（胜率%s）\n请预测比赛结果...",
    homeTeam, homeWinRate, awayTeam, awayWinRate
);
```

## 四、数据库设计思路

### 4.1 表结构设计

**设计原则：**
- 遵循第三范式，减少数据冗余
- 合理设置主键和外键，保证数据完整性
- 为常用查询字段建立索引，提高查询效率

**核心表设计：**
- **emp表**：存储用户基本信息，username设为唯一键
- **user_stats表**：存储用户统计数据，通过username关联用户
- **nbaplayerbasicstatistics表**：存储球员基础数据
- **nbateamstats表**：存储球队数据
- **nba_advanced_stats表**：存储球员高阶数据

### 4.2 关系设计

```
emp (1) ←→ (1) user_stats        一对一关系
nbateamstats (1) ←→ (N) nbaplayerbasicstatistics    一对多关系
nbateamstats (1) ←→ (N) nba_advanced_stats          一对多关系
```

## 五、关键技术难点与解决方案

### 5.1 SSE流式响应

**难点**：如何实现AI响应的实时推送？

**解决方案**：
- 后端使用Spring的SseEmitter类创建服务器推送事件
- 设置180秒超时，防止连接长时间占用
- 使用ExecutorService异步处理，避免阻塞主线程
- 前端使用EventSource接收事件，实时渲染内容

### 5.2 跨域问题

**难点**：前后端分离开发时的跨域请求问题。

**解决方案**：
- 后端添加@CrossOrigin注解允许跨域
- 配置CORS过滤器，设置允许的源和方法
- 开发环境使用代理，生产环境部署同源

### 5.3 大数据量渲染

**难点**：球员数据量大时页面渲染慢。

**解决方案**：
- 前端实现分页显示，每页只渲染固定数量
- 使用虚拟滚动技术，只渲染可视区域元素
- 添加加载动画，提升用户体验

### 5.4 AI API限流

**难点**：智谱AI API有调用频率限制。

**解决方案**：
- 实现本地降级响应，当API限流时返回预设答案
- 添加重试机制，延迟后重新请求
- 优化提示词，减少不必要的API调用
